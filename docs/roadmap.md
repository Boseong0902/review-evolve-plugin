# Phase 3-5 설계 로드맵

> Phase 1-2는 구현 완료. 이 문서는 Phase 3-5의 상세 설계를 담고 있습니다.

## Phase 3: 자동 캡처 고도화 + 노이즈 필터링

### 자동 캡처 고도화
현재 PostToolUse 훅은 두 가지 방식으로 리뷰 파일을 판별합니다:
1. `.review-evolve/claude-review/` 경로 감지 (PreToolUse 훅이 지시한 JSON 저장)
2. `## Critical/Major/Minor/Positive` 마커 패턴 매칭 (폴백)

Phase 3에서는:

1. **SubagentStop 이벤트 활용**: code-reviewer 에이전트가 종료될 때 캡처하여 더 정확한 트리거
2. **리뷰 메타데이터 추출**: 리뷰 대상 커밋 해시, 브랜치, PR 번호 등 연결
3. **변경 영향 범위 분석**: 변경된 파일이 import하는 모듈까지 컨텍스트 확장

### 노이즈 필터링 상세 설계

**자동 삭제 규칙:**
```
- pending + 1회 + 한쪽만 + 미승인 → 30일 후 자동 삭제
- severity: Minor + frequency: 1 + confidence: pending → 다음 리뷰 미발견 시 삭제
- 동일 패턴의 frequency가 0으로 30일간 유지 시 → archived 섹션으로 이동
```

**패턴 병합:**
- 의미적으로 유사한 패턴 자동 감지 (키워드 유사도)
- 사용자에게 병합 제안: "이 두 패턴을 합칠까요?"

## Phase 4: 컨텍스트 예산 관리 고도화 + /review-dashboard

### 컨텍스트 예산 고도화

현재 구현: 문자 수 기반 토큰 추정 (chars / 4)

Phase 4에서는:
1. **정확한 토큰 카운팅**: tiktoken 라이브러리 활용
2. **동적 예산 조절**: 리뷰 대상 코드 크기에 따라 예산 비율 조절
3. **패턴 압축**: 유사 패턴을 그룹화하여 하나의 요약으로 주입

### /review-dashboard 스킬

축적된 패턴을 조회하고 관리하는 대시보드 스킬:


## Phase 5: 실제 프로젝트 검증

### 검증 계획
slunchFactory 프로젝트에서 3회 이상 사이클을 돌려 검증:

1. **Cycle 1**: 기본 동작 확인
   - 코드 변경 → OMC review → 캡처 확인
   - /review-evolve → 교차검증 → 패턴 축적
   - 다음 리뷰에서 패턴 주입 확인

2. **Cycle 2**: 패턴 재발견 + 승격
   - 같은 유형의 코드 변경 → 기존 패턴 재발견 확인
   - frequency 증가 + confidence 승격 확인
   - Codex가 Claude와 다른 이슈 발견하는지 확인

3. **Cycle 3**: 축적 효과 측정
   - 3번째 리뷰에서 이전 패턴이 참고되었는지 확인
   - 리뷰 품질 변화 정성적 평가
   - 토큰 예산 관리 동작 확인

### 측정 지표
- 패턴 축적 수 (사이클당)
- 교차검증 일치율 (Both / Total)
- 재발견율 (이전 패턴이 다시 나타나는 비율)
- 주입된 패턴이 실제 리뷰에 영향을 준 비율

### 예상 결과
- 3회 사이클 후 리뷰 패턴 10-15개 축적
- 교차검증 일치율 30-50% (다른 모델이므로 완전 일치는 기대하지 않음)
- 재발견율 증가 추세 (사이클이 반복될수록 리뷰어가 기존 패턴 참고)
